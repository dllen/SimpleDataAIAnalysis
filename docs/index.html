工作流

三步完成一次真实数据分析

不需要配置复杂数据源，不需要手写 SQL，上传文件后即可开始对话式分析。

Step 01

上传文件

支持 CSV、Excel、JSON。系统自动解析字段、推断类型，并检查是否存在缺失值、异常值或格式问题。

Step 02

AI 分析与清洗

基于 Spring AI 生成 SQL，由 DuckDB 执行；发现脏数据时会给出清洗建议，并可回滚保存清洗前后结果。

Step 03

查看结论与图表

不只是返回表格，还会生成自然语言结论、可视化图表、推荐追问，支持导出或继续深挖分析。

功能说明

为“普通人分析数据”设计的系统能力

当前系统已具备从上传、清洗、分析到可视化的完整闭环，并持续向 SaaS 体验演进。

多格式上传

支持 CSV、Excel、JSON，自动识别字段与基础类型，减少用户手动整理成本。

自然语言分析

用户用中文提问，系统自动生成 SQL 并执行，再返回自然语言答案，而不是只展示原始查询结果。

两步法 Text-to-SQL

LLM 生成 SQL，DuckDB 执行查询，再由 LLM 将结果翻译成用户容易理解的结论。

数据可视化

自动生成图表，例如柱状图、饼图等，帮助用户快速理解数据分布和趋势。

数据清洗建议

自动扫描缺失值、异常值、重复数据与列名问题，给出可执行清洗建议，并支持保存或回滚。

多轮对话

支持上下文连续问答，用户可以在上一轮结论基础上继续追问，获得更深层分析。

流式响应

基于 SSE 推送分析结果，减少等待感，让对话体验更像实时助手。

用户隔离与安全

JWT 认证、多用户数据隔离、SQL 只读沙箱，避免越权访问和危险执行。

安全说明

我们重视数据安全与使用边界

对于表格数据类产品，用户最关心的问题不是“功能多不多”，而是“敢不敢传”。因此系统把安全设计当成核心能力，而不是附加功能。

多用户数据隔离

不同用户的数据在存储和查询层面保持隔离，避免共享或误访问他人文件。

SQL 只读沙箱

分析阶段限制为只读执行边界，降低误删数据或执行危险语句的风险。

清洗可回滚

自动检测到脏数据后，清洗执行前可备份，失败后回滚，避免原始数据丢失。

最小权限原则

系统按分析需要执行最小权限操作，不做无关写入，也不额外暴露内部数据。

技术栈

清晰、可演进、适合持续交付

系统采用主流成熟技术栈，兼顾开发效率、运行性能和后续 SaaS 扩展能力。

后端

Java 17 + Spring Boot 3，Spring AI 接入兼容 OpenAI 接口的大模型，Spring Security + JWT 负责认证。

分析引擎

DuckDB 作为嵌入式分析引擎，支持快速执行 SQL、处理文件型数据查询，并具备列式分析优势。

前端

React + TypeScript + Vite，状态管理、HTTP 请求与可视化组件清晰分离，便于迭代界面与交互。

可视化与交互

ECharts 提供图表能力，Zustand 管理会话状态，Ant Design 提供一致的组件体验。

支持文件

路线图

从可用产品，走向更好用的 SaaS

当前系统已经具备核心分析能力；接下来的重点是降低使用门槛、提升信任感和构建付费闭环。

Phase 1：核心体验版

优先完成游客演示体验、上传分析、基础可视化、登录注册和最小额度体系，让用户先完成第一次真实分析。

Phase 2：助手感增强版

增加自动推荐问题、AI 结论摘要、自动清洗建议、会话保存与继续追问，提升留存和付费转化。

Phase 3：留存与增长版

上线场景模板库、结果分享/导出、新手引导、基础运营看板，形成更稳定的增长和商业化闭环。

常见问题

关于这个项目的几个常见问题

这个系统适合谁用？

适合经常处理 Excel、但不会也不想学 SQL/BI 的用户，例如运营、财务、学生、副业者和内容创作者。

用户上传的数据会被用于训练模型吗？

当前设计强调用户数据隔离与安全边界，系统按分析需要执行最小权限操作，不做超出分析用途的数据滥用。

如果数据有脏值怎么办？

系统会自动检测缺失值、异常值、重复数据和列名问题，并给出清洗建议；清洗执行前可备份，失败可回滚。

后续会支持哪些扩展能力？

后续会围绕场景模板、结果导出、会话保存、模型切换、运营看板等能力继续演进，提升长期使用价值。
